# PG: Naming (for builders)

Source: Paul Graham — *Change Your Name*.

Adopt **PG's specific, opinionated stance** — not the generic "brainstorm and
check availability" that any model defaults to. PG's view of naming is contrarian
in two ways that will actually change what you do.

## PG's perspective (lead with this)

- **Get the `.com`, or change the name.** This is close to a hard rule, not a
  nice-to-have. A startup/project without the `.com` of its name "signals
  weakness" and marginality. Don't settle for the `.io`/`.app`/`.dev`
  workaround — if you can't get the clean name, **pick a different name.**
- **Names barely matter — so stop agonizing.** "Almost any word or word pair that
  is not an obviously bad name is sufficiently good." The instinct to find a
  *clever, perfect* name is mostly wasted effort. Cleverness is nearly worthless;
  *owning a clean, available, not-bad name* is what counts.
- **Naming is a separate skill you're probably bad at** — and that's unrelated to
  your ability to build. Treat it as a quick, almost mechanical task (a good name
  is findable in ~20 minutes), or get help. Don't dignify it as deep creative
  work.

## Apply this when naming

- Generate **10+** candidates fast. The bar is "not obviously bad," not "clever."
- **Verify real availability** before committing: the `.com`, the package
  registry (npm/PyPI/crates), the GitHub org/repo. Don't assume taken — check.
- If the clean name's `.com` is gone and you can't get it cheaply, **change the
  name** rather than accept a worse TLD or a contorted spelling.
- **Timebox to ~20 minutes.** Then pick an available, not-bad name and build.

## Naming review (STOP and check)

1. **Own the clean name?** Is the `.com` (and package/repo) actually available?
2. **Settling for a worse TLD/spelling?** → change the name instead.
3. **Agonizing over cleverness?** The bar is not-bad + available, not brilliant.
4. **Over time-boxed?** If this is taking more than ~20 minutes, just pick one.

## Pushback patterns (apply to yourself)

- "This clever name is perfect" → **cleverness is nearly worthless; is the `.com`
  free?** If not, it's not perfect.
- "I'll just use the `.io` / add a word to get the domain" → **PG says change the
  name.** A contorted name or weak TLD signals marginality.
- "I need more time to find the right name" → **~20 minutes.** Generate, check,
  pick, move on.

## Red flags

- Settling for a non-`.com` or a misspelled/hyphenated name to get a domain.
- Treating naming as deep creative work and burning hours on it.
- Picking a clever name without checking availability.
- Blocking the build on the "perfect" name.

## Anchor (PG)

- "If you don't have the .com of your name, you should probably change your name."
- "Almost any word or word pair that is not an obviously bad name is sufficiently
  good."
- Naming is a separate skill — a good name can be found "in a 20 minute office
  hour slot."
