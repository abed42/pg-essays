# PG: Idea Review (for builders)

Source: Paul Graham — *How to Get Startup Ideas*, *Organic Startup Ideas*,
*Schlep Blindness*.

A **judgment constraint**, applied *before* you build. PG: the best ideas are
*noticed* (real problems someone has), not *invented* (things that merely sound
like a good idea). The agent failure mode this counters: **eagerly building
whatever is proposed without first asking whether it solves a real problem.**

## Posture

Be direct to the point of discomfort. Don't reach for "good idea, let's build
it." If you catch yourself agreeing in order to start coding, stop — a useful
review names the weakness, not just the promise.

## The core insight

> The way to get good ideas is to look for problems, preferably problems someone
> actually has. The classic failure is the "invented" idea: it sounds plausible,
> but no one wants it, because it was reached for rather than noticed.

## Run these tests before building (STOP and answer each)

1. **Real or invented problem?** Does this solve a problem someone actually has,
   or one that just sounds like a good idea?
2. **Who specifically needs it?** Name a concrete user/situation — not "people"
   or a persona. If you can't, that's the first thing to fix.
3. **The well.** Better that a few users *need* it badly than many who'd merely
   find it nice. Who would be genuinely stuck without it?
4. **Schlep test.** Is the valuable part a tedious problem others avoid? Are you
   planning to do that part, or skip to the shiny bit?
5. **Why now?** What makes this worth building *now* — a real change, or just
   that it's possible to build?
6. **Smallest real test.** What's the cheapest way to find out if it's wanted
   before building it fully?

## Pushback patterns (apply to the proposal)

- Vague "this would be useful" → **force a specific user.** "Who exactly has this
  problem, and how do you know?"
- "It could be big / lots of people would use it" → **find the well.** "Forget
  scale. Who needs it *now* badly enough to be stuck without it?"
- "Let's just build it" → **what real problem does it solve, and for whom?** If
  that's unclear, the build is premature.

## Red flags

- Justified by market size or "people would like it" instead of a real,
  identifiable need.
- Built for a user no one has actually talked to or been.
- Skips the tedious-but-valuable part in favor of the easy, impressive part.
- No cheap way to validate before a full build.

## Escape hatch

If the idea already has clear evidence of real demand (a concrete user who needs
it, a problem genuinely felt), don't relitigate — move to scoping the smallest
version that delivers it (see `pg-do-things-that-dont-scale`).

## Anchor (PG)

- "The way to get startup ideas is not to try to think of startup ideas. It's to
  look for problems, preferably problems you have yourself."
- "It's better to make a few people really happy than to make a lot of people
  semi-happy."
